## Shared Protobuf Rules
- Every service has its own .proto in /proto/{service}/v1/
- All services import from this shared location via go.mod replace directive
- Code generation command: `buf generate` from /proto root
- Changing one .proto → run buf generate → all consumers updated automatically
- buf.gen.yaml must generate: go + grpc stubs into /services/{name}/gen/

## Per-Service Go Module Template
Apply this EXACT structure to every service:

```
services/{name}/
├── cmd/
│   └── main.go          # Entry: load config → connect deps → start gRPC server
├── internal/
│   ├── handler/         # gRPC method implementations (thin — call service layer only)
│   ├── service/         # Business logic (no DB access here)
│   ├── repository/      # All DB + Redis queries (pgx/v5, raw SQL)
│   ├── middleware/       # gRPC interceptors: auth, logging, metrics, recovery
│   └── model/           # Domain structs (NOT proto types)
├── config/
│   └── config.go        # Env loading via viper, returns typed Config struct
├── gen/                 # Generated proto stubs (git-ignored, generated by buf)
├── migrations/          # SQL migration files (001_init.up.sql, etc.)
├── Dockerfile
├── .env.example
└── README.md            # Service-level: responsibilities, env vars, gRPC methods
```

## Services — Full Specification

### gateway
- Role: Public REST API entry point, routes to internal gRPC services
- Routes:
  - POST /auth/register → auth.Register
  - POST /auth/login    → auth.Login
  - POST /auth/refresh  → auth.Refresh
  - POST /auth/logout   → auth.Logout
  - GET/POST/PUT/DELETE /users/**     → user service
  - GET/PUT /profiles/**              → profile service
  - GET/POST/PUT/DELETE /questions/** → question-bank service
  - GET/POST/PUT/DELETE /exams/**     → exam service
  - POST /submissions/**              → submission service
  - GET /results/**                   → result service
- Middleware: JWT validation, request logging, rate limiting, CORS
- Health: GET /healthz

### auth
- gRPC methods: Register, Login, Logout, RefreshToken, ValidateToken
- JWT: access token 15min, refresh token 7d, stored in Redis
- Password: bcrypt cost 12
- Redis key pattern: session:{user_id}:{token_id}

### user
- gRPC methods: CreateUser, GetUser, UpdateUser, DeleteUser, ListUsers, ChangeRole
- Roles: admin · teacher · student
- Soft delete (deleted_at timestamp)

### profile
- gRPC methods: GetProfile, UpdateProfile, UploadAvatar
- Avatar: store URL only (not binary)
- One profile per user (FK to user_id)

### question-bank
- gRPC methods: CreateQuestion, GetQuestion, UpdateQuestion, DeleteQuestion,
                ListQuestions, SearchQuestions, BulkImport
- Question types: multiple_choice · true_false · short_answer
- Fields: content, options[], correct_answer, difficulty (1-5), subject, tags[], created_by

### exam
- gRPC methods: CreateExam, PublishExam, GetExam, ListExams, UpdateExam,
                DeleteExam, RandomizeQuestions
- RandomizeQuestions: select N questions from bank filtered by subject/difficulty/tags
- Cache: published exam by ID → Redis, TTL 10 min
  Key: exam:published:{exam_id}
- Fields: title, description, duration_minutes, max_attempts, show_result_immediately

### submission
- gRPC methods: StartSubmission, SubmitAnswer, FinalizeSubmission, GetSubmission
- Auto-grading: compare answers against correct_answer, calculate score
- On finalize: publish RabbitMQ event → notification service
  Exchange: learnify.events
  Routing key: submission.completed
  Payload: { submission_id, student_id, exam_id, score, passed }

### result
- gRPC methods: GetResult, ListResultsByStudent, ListResultsByExam,
                GenerateReport, GetClassPerformance
- Stores: score, percentage, passed (bool), time_taken_seconds, per_question breakdown

### notification
- RabbitMQ consumer only — no gRPC server
- Consumes: submission.completed events
- Actions: (stub for now) log notification, future: email/websocket
- Queue: learnify.notification.queue bound to learnify.events exchange
